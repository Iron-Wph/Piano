# 钢琴乐谱到虚拟手教学视频：MVP 方案

## 1. 文档目的

本文档定义一个以“最快验证真实需求”为目标的 MVP。该版本不建设网页或移动端 UI，不追求真人手、3D 高保真动作或实时跟弹，而是通过本地命令行工具完成核心链路验证。

研究依据参见：[deep-research-report.md](./deep-research-report.md)。

## 2. 一句话定义

将 MIDI 或 MusicXML 乐谱自动转换成包含虚拟键盘、左右手、手指编号、按键高亮和同步音频的教学视频，并允许用户通过配置文件修改指法后重新生成。

## 3. MVP 要验证的问题

MVP 不是为了证明“能播放 MIDI”，而是验证以下三个产品假设：

1. 钢琴教师或教学内容创作者是否需要把自有乐谱快速转换为手部教学视频。
2. 虚拟手动画即使不够逼真，只要按键、指法和时序清晰，是否已经具备教学价值。
3. 用户是否愿意修改自动指法并重新生成视频，以及这种方式是否比真人录制更高效。

## 4. 首批目标用户

### 4.1 核心用户

- 需要制作课后示范视频的个人钢琴教师。
- 需要批量制作不同速度、左右手版本内容的培训机构。
- 使用 MIDI/MusicXML 制作教学内容的钢琴自媒体创作者。

### 4.2 暂不作为首批核心用户

- 只需要系统课程的零基础消费者。
- 需要实时演奏评分的学生。
- 需要 AR/MR 或智能钢琴硬件联动的用户。
- 只拥有纸质、手写或复杂扫描乐谱的用户。

## 5. 核心价值

当前教师制作一段手部示范视频，通常需要练习、录制、剪辑、纠错和重新录制。MVP 要把这一过程缩短为：

```text
准备 MIDI/MusicXML
→ 自动分析指法
→ 检查并修改配置
→ 生成多个教学视频版本
```

用户最终购买的不是“虚拟手效果”，而是以下能力：

- 任意数字乐谱可以快速转成教学内容。
- 指法、速度和练习范围可以修改。
- 同一首曲目可以批量生成不同版本。
- 不需要真人出镜、录制设备和重复剪辑。

## 6. 最小产品范围

### 6.1 P0：必须完成

| 模块 | MVP 能力 |
|---|---|
| 输入 | 导入标准 MIDI、MusicXML 文件 |
| 乐谱解析 | 提取音高、起止时间、声部、谱表、速度、拍号和踏板基础信息 |
| 左右手分配 | MusicXML 优先使用谱表信息；MIDI 使用音域与连续性规则推断 |
| 指法生成 | 使用规则法或动态规划生成 1–5 指编号 |
| 人工修改 | 导出 YAML/JSON 配置，允许用户修改左右手、指法、速度和小节范围 |
| 动画 | 生成二维或 2.5D 示意手、手指编号、按键高亮和键盘动画 |
| 时间控制 | 支持原速、指定 BPM、倍率调速和选定片段 |
| 音频 | 使用 MIDI 合成钢琴音色，并与视频事件时间轴同步 |
| 输出 | 输出 MP4、标准化时间轴 JSON、指法 CSV 和运行报告 |
| 质量检查 | 检查未分配音符、非法指号、超大手跨度和音视频时长差异 |

### 6.2 P1：验证有效后再做

- 左手或右手单独版本。
- 自动循环指定小节。
- 俯视与斜视两种镜头模板。
- MusicXML 指法标记回写。
- 多套手型偏好，例如儿童、小手、成人。
- 批量处理目录内的多个曲目。

### 6.3 明确不进入本次 MVP

- 网页、桌面或移动端图形界面。
- 用户系统、云端服务、支付和权限管理。
- PDF、图片、手写谱或简谱识别。
- 真人手视频合成。
- 高保真 3D 手和物理接触仿真。
- 麦克风跟弹、实时评分和实时跟谱。
- 自动语音讲解、课程系统和曲库运营。
- 社区分享及公开视频托管。

## 7. MVP 使用流程

### 7.1 第一次生成

```bash
piano-hand analyze examples/song.musicxml --output work/song
piano-hand render work/song/project.yaml
```

系统输出：

```text
work/song/
├── project.yaml
├── timeline.json
├── fingering.csv
├── validation-report.json
└── output.mp4
```

### 7.2 修改后重新生成

用户在 `project.yaml` 或 `fingering.csv` 中修改以下内容：

- 某个音符使用哪只手。
- 某个音符使用哪根手指。
- 视频速度。
- 需要导出的起止小节。
- 是否显示手指编号、按键标签和节拍线。

然后执行：

```bash
piano-hand validate work/song/project.yaml
piano-hand render work/song/project.yaml
```

## 8. 最快验证方案

### 8.1 验证样本

准备 20 首具有合法使用权的测试曲目：

- 5 首单手或非常简单的入门曲。
- 10 首双手初级曲。
- 5 首包含和弦、跨指或较大跳跃的中级片段。
- MIDI 与 MusicXML 各至少覆盖 8 首。

### 8.2 目标访谈用户

建议邀请 5–10 名钢琴教师或教学创作者参加测试。每位用户至少完成：

1. 选择一首自己熟悉的曲目。
2. 查看系统自动生成的指法与视频。
3. 修改至少一处指法或速度。
4. 重新生成视频。
5. 判断是否愿意把结果用于真实授课、课后练习或内容发布。

### 8.3 核心验证指标

| 指标 | 建议通过线 |
|---|---:|
| 支持样本成功生成率 | ≥ 90% |
| 音符与按键时序正确率 | ≥ 98% |
| 无效或不可执行指法比例 | ≤ 5% |
| 一首 3 分钟曲目从导入到首次视频的人工操作时间 | ≤ 10 分钟 |
| 修改指法后重新生成所需人工操作时间 | ≤ 5 分钟 |
| 认为视频“可用于辅助教学”的测试用户比例 | ≥ 60% |
| 明确愿意继续试用或提供第二首曲目的用户比例 | ≥ 40% |

这些数值是产品决策阈值，不是已经验证的事实。

## 9. MVP 成功与失败判定

### 9.1 继续开发

满足以下条件时，进入带基础编辑界面的下一阶段：

- 多数教师认为虚拟手示意足以解释手指与按键关系。
- 自动指法即使需要修改，仍明显快于真人录制。
- 用户主动提出批量生成、模板或更便捷编辑需求。
- 失败主要来自操作方式，而不是核心教学价值不足。

### 9.2 调整方向

出现以下情况时，应调整算法或目标用户：

- 教师认可视频，但自动指法修改量过大。
- 初级曲有效，中级曲无法稳定生成。
- 教师不需要完整视频，但需要指法时间轴、单手片段或课堂演示素材。

### 9.3 停止投入

出现以下情况时，不应直接进入 UI 或 3D 手开发：

- 用户认为虚拟手无法提供比瀑布流更高的教学价值。
- 用户不愿意准备 MIDI/MusicXML。
- 视频生成并未显著降低教师的制作时间。
- 主要需求实际是实时评分或课程内容，而不是教学视频生产。

## 10. 竞争力与壁垒

### 10.1 初期差异化

- 不是单纯的 MIDI 瀑布流，而是明确展示左右手和手指。
- 不是固定课程，而是处理用户自己的乐谱。
- 不是不可控的生成视频，而是可修改的结构化时间轴。
- 不是面向演奏展示，而是面向教学内容生产。

### 10.2 可积累壁垒

早期技术本身不是强壁垒。真正可积累的资产包括：

- 教师修改后的高质量指法数据。
- 不同手型、水平和教学流派的指法偏好。
- 常见错误指法和难点片段。
- 乐谱、指法、手位与动画之间的结构化映射。
- 可复用的视频模板和教学内容生产流程。

## 11. 建议周期与人员

这是用于需求验证的 CLI 原型，不是研究报告中所述的完整教师可用产品。

建议配置：

- 1 名 Python/算法工程师：解析、指法和数据结构。
- 1 名图形/多媒体工程师：动画、音频和 FFmpeg 输出。
- 产品负责人或钢琴教师顾问按需参与验收。

建议周期：4–6 周。

| 周期 | 目标 |
|---|---|
| 第 1 周 | MIDI/MusicXML 解析、统一事件模型、测试样本 |
| 第 2 周 | 左右手与指法基线、配置文件导出 |
| 第 3 周 | 键盘、按键和虚拟手帧渲染 |
| 第 4 周 | 音视频合成、CLI、校验报告 |
| 第 5 周 | 曲目兼容性修复、教师测试 |
| 第 6 周 | 根据反馈迭代并作继续/停止决策 |

## 12. MVP 交付物

- 可运行的本地 CLI 工具。
- 20 首内部合法测试样本及预期结果。
- 至少 3 个完整示例项目。
- 自动生成的视频和结构化中间文件。
- 安装与使用说明。
- 测试报告。
- 教师反馈记录模板。
- 一份基于指标的继续、调整或停止建议。
